
What I found:
- This is not a UI-only issue. “Mammoth Graveyard” is not hardcoded anywhere; the bad result is being generated by the rapid identification pipeline, saved into recent scans, then rehydrated after refresh.
- The rapid queue still allows multiple background workers with very small delays, which reintroduces the old “same result across many cards” failure mode.
- The anomaly pause is only stored in memory. After a refresh, the app starts fresh and auto-resumes the remaining queue, so the broken batch keeps processing.
- Z.AI OCR text is being collected in the queue, but `rapid-card-identify` currently ignores `ocrText`, so the main identifier is still guessing from image alone.

Plan:
1. Re-serialize rapid scan processing
- Force the rapid identify pipeline back to a single worker.
- Restore a minimum gap between identify requests so cards are processed one at a time in the background.
- Keep capture fast; only the AI/price stage becomes serialized.

2. Persist anomaly pause across refresh
- Save a `paused-by-anomaly` flag outside transient store state.
- Make startup auto-resume refuse to restart when that flag is set.
- Clear the flag only when the user explicitly resumes.

3. Stop poisoned batches earlier
- When repeated-name detection hits the threshold, stop the queue before more bad cards are written into recent scans.
- Mark remaining queued items as paused/error instead of silently continuing.
- Add a safe recovery path to clear the bad local rapid-scan batch without touching saved library cards.

4. Actually use OCR evidence in rapid identify
- Update `supabase/functions/rapid-card-identify/index.ts` to accept `ocrText`.
- Feed OCR text into the prompt with priority on title, set code, and card number.
- Keep official-name verification after OCR-backed extraction.

5. Make the failure visible in rapid scan
- Show a clear anomaly-paused banner in `RapidScanCamera`.
- Add Resume and Clear Recent Rapid Scans actions so a refresh cannot hide the broken state.

Files:
- `src/lib/queueProcessor.ts` — single-worker rapid queue, persisted anomaly pause, earlier hard stop
- `src/App.tsx` — ensure startup resume respects persisted anomaly state if needed
- `src/components/scanner/RapidScanCamera.tsx` — paused banner and recovery controls
- `supabase/functions/rapid-card-identify/index.ts` — consume OCR text and tighten evidence flow

What stays unchanged:
- No database/schema changes
- No saved collection/library records are deleted
- The 500 recent-scan cap stays as-is; this bug is the corrupted rapid-scan batch, not the old list limit
